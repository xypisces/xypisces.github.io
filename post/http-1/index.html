前端性能优化原理与实践

发布于 2018-04-16 | 分类于 Web | 6分钟 | 1529字数

url从输入到页面到底发生了什么，这几乎是面试必考题，当你回答出来了，然后面试官问你怎么针对性优化，你还能答得上来么？

url->页面的生成过程

  • DNS查询,返回IP地址
  • TCP三次握手，应用层将数据传到传输层，TCP指明端口号，下发到网络层的IP协议确定IP地址，下发到数据链路层封装数据帧，最后是物理层面的传输
  • 如果有HTTPS则还会进行TLS握手，然后进行真正的传输
  • 如果有负载均衡的服务器，则将请求分发到多个服务器，
  • 假设服务器响应一个文件，浏览器会先判断状态码，200则进行解析，300则进行重定向，400或500则报错
  • 如果是gizp等格式则会解压一下，然后通过文件编码去解析
  • HTML文件则会构建DOM树，CSS树，遇到JS文件会先判断async还是defer进行下载，如果什么都没有则会阻塞渲染直到JS执行
  • HTML完全加载解析后触发DOMContentLoaded树
  • CSS树和DOM树构建完成生成render树
  • 最后调用GPU绘制，合成图层，显示到屏幕中

图片层面的优化

针对不同场景的图片，使用不同的格式能对网站可以有很大的提升，下面的总结的是各个格式的适用场景

  • JPEG/JPG: 有损压缩，适用于大图，色彩丰富的图片
  • PNG: 无损压缩，支持透明，适用于小图Logo等
  • SVG: 文本文件，体积小，兼容性好，适用于简单图形icon等
  • Base64: 文本文件，与svg一样，小于2kb的图片可以使用base64，减少请求
  • WebP: 全能型，但是兼容性不好

HTTP缓存机制

HTTP 缓存是我们日常开发中最为熟悉的一种缓存机制。它又分为强缓存和协商缓存。优先级较高的是强缓存，在命中强缓存失败的情况下，才会走协商缓存。

强缓存

  • expires

expires: Wed, 11 Sep 2019 16:12:18 GMT

expires是一个时间戳，当本地时间小于expries设置的时间，则会去缓存中取，缺点就是如果直接修改本地时间，则expries就会达不到想要预期，因此诞生了cache-control

  • cache-control

cache-control: max-age=31536000

cache-control优先级比较高，max-age是一个时间长度，在这里时间长度以内的请求的资源都从缓存中取

协商缓存

  • Last-Modified

首次请求后会返回Last-Modified: Fri, 27 Oct 2017 06:35:57 GMT ，随后发起请求后回带上If-Modified-Since: Fri, 27 Oct 2017 06:35:57 GMT，当服务端判断该时间戳和资源在服务器最后修改的时间是否一致，从而判断是否发生变化。

存在的缺点就是1.编辑了文件，但是内容没变，服务端无法识别，只通过时间修改判断，从而会被重新请求。2.修改过快也会导致服务端感知不到（识别秒级以上的差异）

  • Etag

首次请求后会返回ETag: W/"2a3b-1602480f459" ，随后发起请求后回带上If-None-Match: W/"2a3b-1602480f459"，Etag 的生成过程需要服务器额外付出开销，会影响服务端的性能，这是它的弊端。因此启用 Etag 需要我们审时度势。

选择合适的缓存策略

  • 对于某些不需要缓存的资源，可以使用 Cache-control: no-store，表示该资源不需要缓存
  • 对于频繁变动的资源，可以使用 Cache-Control: no-cache 并配合 ETag 使用，表示该资源已被缓存，但是每次都会发送请求询问资源是否更新。
  • 对于代码文件来说，通常使用 Cache-Control: max-age=31536000 并配合策略缓存使用，然后对文件进行指纹处理，一旦文件名变动就会立刻下载新的文件。

浏览器渲染过程

解析HTML->计算样式->计算图层布局->绘制图层->GUP渲染得到页面

优化点：

  • 由于css选择器是从右往左边开始匹配，因此要避免过深的嵌套，少用标签选择器，会全局搜索匹配，浪费性能，多用类选择器
  • HTML，CSS，JS都会堵塞，从上至下的进行渲染，因此css尽量放到头部，js放到尾部
  • js有三种加载方式:正常加载，defer加载(异步加载，等文档解析完全后再执行。当脚本依赖于 DOM 元素和其它脚本的执行结果时，我们会选用 defer)， async加载(异步加载，加载结束就执行，一般当我们的脚本与 DOM 元素和其它脚本之间的依赖关系不强时，我们会选用 async)

重绘与回流

  • 重绘：当节点需要更改外观而不会影响布局的，如改变颜色
  • 回流：布局或者几何属性需要改变就称为回流，如改变大小

重绘不一定会引发回流，但是回流一定会引发重绘。下面几个操作就会导致回流和重绘

  • 改变window大小
  • 改变字体大小
  • 添加或删除样式
  • 文字改变
  • 定位或者浮动
  • 盒模型修改

浏览器一帧中发生了什么？
1.Event loop执行完Microtasks会判断document是否需要更新
2.判断resize或者scroll时间
3.判断是否触发了media query
4.更新动画并且发送事件
5.判断是否有全屏操作
6.执行 requestAnimationFrame 回调
7.执行 intersectionObserver回调判断元素是否可见
8.更新界面

如何渲染几万条数据并不卡住界面

setTimeout(()=>{
  const total = 100000
  const once = 20
  const loopCount = total / once
  let countOfRender = 0
  let ul = document.querySelector('ul')
  function add(){
    const fragment = document.createDocumentFragment()
    for(let i=0; i<once; i++){
      const li = document.createElement('li')
      li.innerText = Marh.floor(Math.random() * total)
      fragment.appendChild(li)
    }
    ul.appendChild(fragment)
    countOfRender += 1
    loop()
  }
  function loop(){
    if(countOfRender < loopCount) {
      window.requestAnimationFrame(add)
    }
  }
},0)
